## Context

当前主窗口实现集中在 `src/views/MainView.vue`，它只承担“快捷入口 + 最近笔记列表”两块内容，没有统一的页面壳层；`CanvasView`、`SearchView`、`SettingsView` 虽然已经在主窗口内通过 `ui` store 路由切换，但每个页面仍然维持各自独立的页头与布局，导致页面之间缺少一致的桌面工作台体验。

`docs/steno-layout-v2.html` 给出了新的主窗口原型：左侧为可折叠导航，顶部为自定义标题栏和搜索入口，中间为内容区。该原型还引入了多个尚未实现的模块菜单项，意味着主窗口除了展示现有能力，还必须成为未来能力的稳定宿主。与此同时，当前产品已经明确区分“主窗口内整理/编辑”和“全局浮窗速记”两条工作流，因此新建笔记入口不能再复用 quicknote 浮窗。

## Goals / Non-Goals

**Goals:**

- 建立一个统一的主窗口工作台壳层，让笔记列表、主窗口编辑页、画布、搜索、设置和未来模块共享同一套标题栏与侧边导航。
- 明确定义“新建笔记”与“新建速记”的职责边界：前者进入主窗口编辑页，后者继续使用 quicknote 浮窗。
- 在不改变画布核心交互的前提下，将画布页面改造成工作台中的一个内容页。
- 为“粘贴板、待办、截图、OCR、翻译”提供统一占位页，保证信息架构和导航在功能未完成前就稳定下来。
- 保持现有 Tauri 主窗口单实例模型，避免为了页面改版重新引入多个主窗口类窗口。

**Non-Goals:**

- 不在本次变更中实现粘贴板、待办、截图、OCR、翻译的真实业务逻辑、数据模型或系统权限集成。
- 不重写 quicknote 浮窗、置顶便签、Zen 写作或现有 SQLite 数据模型。
- 不改变画布的拖拽、缩放、过滤等领域行为，只调整其在主窗口中的展示结构。

## Decisions

### Decision 1: 以“工作台壳层 + 内容页插槽”重构主窗口

主窗口将拆成统一壳层和内容页两层。壳层负责自定义标题栏、侧边栏、页标题、页面级操作区和窗口控制；内容页只关注自身业务。这样可以把 `MainView` 从“整页实现”收缩为“笔记列表内容页”，并让 `CanvasView`、`SearchView`、`SettingsView` 和未来占位页共享同一套外壳。

替代方案是继续让每个页面各自维护完整布局，只在视觉上模仿原型。这种做法会把导航、窗口控制、页面标题和搜索入口复制到多个页面，后续维护成本会快速上升，因此不采用。

### Decision 2: 在主窗口路由中新增笔记编辑页，而不是复用 Zen 页面

主窗口需要一个标准编辑页来承接“新建笔记”和“从列表打开笔记”的日常编辑流。该页面保留普通工作台上下文，适合浏览、编辑、返回列表和继续切换其他模块。Zen 保持沉浸式写作定位，作为显式高级模式保留，而不是默认编辑页。

替代方案是直接把 Zen 当作主窗口编辑页，但这会混淆“常规整理”与“无干扰写作”两种模式，也无法匹配原型里的列表/编辑双态工作台结构，因此不采用。

### Decision 3: 自定义标题栏采用“拖拽区域 + 交互控件白名单”实现

标题栏将使用 Tauri 支持的拖拽区域能力，默认允许在品牌区和头部空白区域拖动窗口；搜索框、返回按钮、窗口控制按钮等可交互元素必须显式排除拖拽。这样既能满足原型的自定义视觉，又不会牺牲窗口操作的可用性。

替代方案是保留系统原生标题栏，仅把原型头部当作普通工具栏。这会直接违背设计目标，也无法统一跨平台视觉，因此不采用。

### Decision 4: 未实现模块统一使用通用占位页组件

“粘贴板、待办、截图、OCR、翻译”不单独各写一套页面逻辑，而是共享一个通用占位页组件，只通过标题、描述和后续说明文案区分。这样可以先稳定导航结构和页面切换逻辑，避免未来功能落地前页面结构反复变化。

替代方案是先不把这些模块放进导航，等功能准备好再加入。但原型和当前需求都明确希望导航先稳定，因此不采用。

### Decision 5: 画布继续使用现有内容组件，只迁移页头与容器布局

画布本身的交互复杂度已经较高，本次只把页面头部和容器布局迁移到工作台壳层中，并让 `Canvas` 组件继续作为主要交互核心存在。这样可以把风险集中在壳层整合和路由改造，而不是重新实现画布业务。

替代方案是借主页面重构同步重写画布工具栏和交互层，但这会把本次改版从“壳层整合”扩张为“重新设计画布产品”，超出当前范围，因此不采用。

## Risks / Trade-offs

- 自定义标题栏跨平台行为差异 -> 在 Windows、macOS、Linux 上分别验证拖拽、双击最大化、窗口控制按钮命中区域，必要时对双击行为做平台降级。
- 主窗口路由扩展会影响现有页面切换逻辑 -> 用 `ui` store 明确区分“工作台页面路由”和“独立窗口模式”，并补充路由级测试。
- 主窗口编辑页与 Zen 并存可能增加复杂度 -> 在导航与操作文案上明确两者职责，普通“编辑”默认走主窗口页，Zen 仅作为附加入口。
- 占位页先上线但没有真实功能 -> 页面文案必须清楚标明“功能规划中”，避免用户误解为异常或缺陷。
- 画布嵌入壳层后可用高度减少 -> 通过壳层布局约束内容区高度，让 `Canvas` 始终占满剩余空间，避免出现双重滚动条。

## Migration Plan

1. 先引入新的主窗口路由模型和统一工作台壳层，但保持 quicknote、sticky、Zen 的窗口标签和入口不变。
2. 新增主窗口编辑页，并把笔记列表中的“新建笔记/编辑笔记”入口切到该页面。
3. 将画布、搜索、设置逐步迁移到工作台壳层中，验证现有功能没有退化。
4. 新增通用占位页并接入侧边导航，把未实现模块路由稳定下来。
5. 回归验证主窗口页面切换、自定义标题栏拖拽、窗口控制、quicknote 入口分流和画布交互。

## Open Questions

- 顶部搜索栏第一版是直接复用现有搜索页跳转，还是在笔记列表中先做即时过滤；当前计划优先保留现有搜索能力并以跳转或聚焦方式接入。
- 主窗口编辑页是否需要在第一版同时支持预览分栏；当前倾向复用现有 `MarkdownEditor` 能力并跟随全局编辑器设置。
- 原型中的底部“置顶粘贴板”条带是否属于本次范围；当前不纳入本次需求，待粘贴板模块设计时单独定义。
