## Context

当前 Steno 的主编辑页 `NoteEditorView` 与 Zen 页 `ZenMode` 都建立在 `MarkdownEditor.vue` 的 `textarea + marked 预览` 模型之上。这个模型适合速记浮窗、便签等轻量场景，但它无法提供“直接在排版结果上编辑”的主写作体验，也没有内建的大纲侧栏、代码态切换约束、悬浮展开入口、折叠阈值和宽度持久化能力。

项目内已经出现两类不同的编辑需求：一类是 quicknote / sticky 这类轻量文本编辑；另一类是主窗口编辑页与 Zen 写作这类结构化、长文档导向的写作体验。用户这次明确要求后者向 `PureMark` 的 Markdown-first 富编辑模式靠拢，同时保持 Steno 现有的 Markdown 存储、自动保存、导出与标签抽取链路不变。

## Goals / Non-Goals

**Goals:**

- 为主编辑页和 Zen 页建立一套共享的 Markdown-first 富编辑内核，默认以排版编辑态工作，而不是源码态。
- 定义并落实三态编辑合同：排版编辑态、只读态、代码态，以及这些模式之间的切换规则。
- 为普通编辑页与 Zen 页提供一致的大纲能力，包括标题树、跳转、拖拽改宽、折叠阈值、重新展开与持久化恢复。
- 在主编辑页底部同时提供只读/编辑切换、代码模式切换和进入 Zen 的入口。
- 保持笔记最终存储仍为 Markdown 字符串，不改动笔记表结构，不破坏现有自动保存、导出和标签抽取逻辑。
- 将主编辑体验与轻量编辑体验解耦，使主窗口编辑页 / Zen 的升级不会把 quicknote、sticky 等现有场景一起拖入重构。

**Non-Goals:**

- 不在本次变更中重写 quicknote 浮窗、StickyNote 或 FloatingEditor 的编辑内核。
- 不在本次变更中改变 `notes` 数据表结构、笔记导出协议或现有自动保存时机。
- 不承诺一次性补齐 PureMark 的全部高级语法能力；首版目标是围绕标题、段落、列表、引用、代码块和基础行内样式建立稳定编辑合同。
- 不在本次变更中新增云同步、协作、评论或跨设备共享能力。

## Decisions

### Decision 1: 主编辑页与 Zen 引入独立的富编辑内核，保留现有 `MarkdownEditor.vue` 供轻量场景继续使用

主窗口编辑页与 Zen 页将不再直接复用 `MarkdownEditor.vue`。我们会新增主编辑器抽象层和富编辑组件，只在主编辑页与 Zen 中挂载。`MarkdownEditor.vue` 继续服务于 quicknote、sticky note 和其他轻量编辑场景，避免一次重构影响全部窗口类型。

这样做的原因是现有 `textarea + preview` 结构无法自然表达“在排版结果上编辑”的交互，也不适合作为大纲、模式切换和布局控制的中心。把主编辑体验拆成单独能力层，可以把复杂度局部化。

替代方案是继续增强 `MarkdownEditor.vue`，在上面叠加 `contenteditable` 或双向 DOM/Markdown 同步。但这种方案会把轻量编辑器推向不稳定的中间态，而且很难规避语法同步和状态分裂问题，因此不采用。

### Decision 2: 采用“Markdown-first 富编辑 + 源码回退”的架构，技术方向参考 PureMark 的 ProseMirror / CodeMirror 分层

新主编辑器采用 Markdown-first 的富编辑思路：默认展示排版后的可编辑内容，内部维护可序列化回 Markdown 的文档状态；源码态则使用独立源码编辑视图承接原始 Markdown 文本。技术上优先采用与 PureMark 相同的分层方向，即 ProseMirror 级富编辑核心 + 源码编辑视图分离的结构，而不是纯预览 DOM 的双向修补。

这样可以同时满足两个目标：一是排版态可直接编辑，二是始终保有可导出、可保存、可抽取标签的 Markdown 规范文本。源码态天然还能作为复杂语法的回退入口。

替代方案一是直接移植 PureMark 全量编辑器与周边能力，但它的工作区、多标签、文件系统和菜单体系超出 Steno 需求，迁移成本过高，因此只参考其编辑器分层与状态治理思路。替代方案二是直接引入传统“左源码右预览”库，但它不能满足默认排版编辑体验，因此不采用。

### Decision 3: 编辑状态使用显式三态枚举，而不是“视图模式 + 只读布尔值”的自由组合

实现层会把主编辑状态建模为显式三态：

- `rich-edit`
- `rich-readonly`
- `source-edit`

这样可以直接表达当前产品合同，避免出现 `source + readonly` 这类用户没有要求、但组合状态上可能被误触发的非法模式。切换规则也更简单：只读按钮只在富编辑态之间切换，代码模式按钮始终进入 `source-edit`；从 `rich-readonly` 进入代码模式时自动退出只读。

替代方案是把状态拆成 `surfaceMode: rich|source` 与 `readOnly: boolean` 两个字段。虽然它看似更“通用”，但会引入产品并不需要的 `source-readonly` 组合和更多守卫逻辑，因此不采用。

### Decision 4: 大纲以共享控制器驱动，普通编辑页用悬浮展开，Zen 页用常驻侧栏

普通编辑页和 Zen 页会共用一套大纲控制器与渲染组件，但外层壳子不同：

- 普通编辑页默认隐藏大纲，只保留悬浮按钮作为展开入口。
- Zen 页默认显示右侧大纲栏，更符合沉浸式长文写作场景。

大纲项以树状标题结构展示，点击后滚动到对应位置。普通编辑页折叠后回到悬浮按钮入口；Zen 页折叠后保留最小入口，再由显式按钮重新展开。

替代方案是两个页面分别维护自己的大纲实现。这会造成跳转、空状态、拖拽阈值和持久化行为分叉，后续维护成本更高，因此不采用。

### Decision 5: 大纲数据从“规范 Markdown 内容”提取，而不是从富编辑 DOM 直接推导

无论当前在排版编辑态还是源码态，大纲都从当前文档的规范 Markdown 文本中提取标题树。这样可以保证：

- 大纲与最终保存内容一致；
- 排版态与源码态共享同一份结构视图；
- 后续测试可以直接对 Markdown 输入和大纲输出做断言，而不依赖具体富编辑 DOM。

替代方案是直接遍历富编辑器内部文档树或 DOM。这会把大纲实现强绑定到单一编辑内核，并提高源码态与富编辑态切换时的同步复杂度，因此不采用。

### Decision 6: 侧栏宽度与展开状态写入设置存储，并按普通编辑页 / Zen 页分别持久化

宽度与折叠状态需要跨应用重启恢复，因此不会只放在组件局部状态或 `localStorage`。本次会扩展 Steno 现有设置存储，为普通编辑页和 Zen 页分别记录：

- 大纲栏宽度
- 大纲栏是否展开 / 折叠

普通编辑页和 Zen 页分别持久化，是因为两种布局的可用空间和使用目标不同，共用一套宽度会让其中一方体验受限。

替代方案一是放在 `localStorage`，但当前项目已有统一 settings 表与 `settings` store，继续分散到浏览器存储会降低一致性和测试可控性。替代方案二是两个页面共用一套宽度值，但难以兼顾工作台与全屏写作场景，因此不采用。

### Decision 7: 当拖拽低于折叠阈值时直接折叠，并在折叠状态禁用拖拽，直到用户显式重新展开

侧栏不会保留“只剩几像素还可以拖”的中间态。实现上会定义两个阈值：

- `collapseThreshold`: 低于该值直接折叠
- `minExpandedWidth`: 侧栏重新展开后的最小可用宽度

折叠后不再渲染拖拽手柄，直到用户通过按钮重新展开。这样可以避免窄栏既难用又容易误拖的问题，也更符合你给出的产品要求。

替代方案是保留极窄栏并允许继续拖拽，但这种状态既不清晰也不稳定，因此不采用。

## Risks / Trade-offs

- [富编辑内核接入复杂度较高] → 通过“只替换主编辑页与 Zen、保留轻量编辑器不动”来收缩风险面，并把源码态作为语法回退通道。
- [Markdown 富编辑与源码态之间可能出现同步差异] → 始终以规范 Markdown 文本作为保存与大纲的单一事实来源，所有模式切换先同步到统一文本值。
- [新增设置键会影响前后端 settings 对齐] → 同步更新前端 `settings` store、Rust 默认设置和相关测试，确保每个新键都有默认值与解码逻辑。
- [Zen 与普通编辑页共享编辑内核后，页面壳差异容易引入条件分支] → 通过“共享编辑核心 + 独立布局壳”拆分责任，把壳层行为留在页面组件，大纲与编辑状态留在共享控制器。
- [顶部上移与圆角调整可能影响当前主工作台间距] → 仅在主编辑卡片容器内调整留白和边界，不改变主窗口其他页面的头部栅格。

## Migration Plan

1. 先新增主编辑器抽象层与大纲控制器，但暂时只在新组件中接线，不改 quicknote / sticky / floating 现有编辑器。
2. 扩展设置存储，加入普通编辑页与 Zen 页的大纲宽度、展开状态默认值与解码逻辑。
3. 用新主编辑器替换 `NoteEditorView` 的编辑主体，接入三态切换、底部按钮、顶部布局和普通编辑页悬浮大纲。
4. 用相同编辑内核替换 `ZenMode` 的正文区域，接入右侧大纲栏、拖拽宽度、折叠阈值与退出返回来源。
5. 最后补齐模式切换、大纲跳转、拖拽/折叠/持久化和视觉布局回归测试，并确认旧轻量编辑场景未受影响。

## Open Questions

- 无。当前产品合同已经通过用户确认锁定；如果实施过程中发现 PureMark 风格内核的接入成本超出预期，允许在不改变产品行为的前提下调整具体库选型，但不得回退到“源码为主、预览为辅”的旧模型。
