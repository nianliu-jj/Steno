## Context

`docs/steno-settings-modal.html` 定义的设置原型是桌面应用弹窗：顶部保留品牌和分类标签，中间是按分类滚动的设置项，底部提供取消、确认、重置和自动保存提示。当前 `src/views/SettingsView.vue` 是独立整页视图，并且 `src/views/MainView.vue` 点击“设置”会通过 `useWindow().openSettings()` 打开或切换到设置窗口，这会打断主窗口中的最近笔记工作流。

现有设置 store 已支持主题、三个快捷键、速记浮窗宽高、失焦关闭延迟、编辑器模式、备份阈值；`useDb().getDataPaths()` 已提供数据目录、数据库文件和备份目录。隐私安全、强调色、便签纸、应用排除名单等原型项目前没有持久化字段，因此本次以界面结构和展示能力为主，不扩大 Rust 数据库 schema。

## Goals / Non-Goals

**Goals:**

- 主窗口点击“设置”后打开 Naive UI `NModal`，而不是离开主窗口或创建新的设置窗口。
- 设置面板视觉和信息架构接近原型：顶部标签、分类内容、底部操作栏、桌面级密度、浅深色都可读。
- 保留现有可持久化设置的真实交互，变更后仍能自动保存并在需要时刷新全局快捷键。
- 将存储路径展示从 `NCode` 风格改为普通 `<code>` 或文本容器，继续避免触发 Naive UI highlight.js 警告。
- 为隐私安全和关于信息提供明确的只读或禁用状态，让用户知道功能边界而不是误以为保存失败。
- 补充测试覆盖弹窗入口、设置分类切换和路径渲染。

**Non-Goals:**

- 不新增数据库 settings 字段，不实现隐私安全、强调色、便签纸、应用排除名单的真实持久化。
- 不重写 `useSettingsStore` 的加载、解码和乐观更新策略。
- 不改变独立 `settings` 窗口模式的兜底行为；该模式仍可以直接渲染设置面板内容。
- 不改造主窗口整体布局、笔记列表数据模型或 Tauri 窗口创建逻辑。

## Decisions

### Decision 1: 设置入口在 `MainView` 内持有模态框状态

主窗口“设置”快捷入口改为设置 `settingsVisible = true`，并在 `MainView` 模板中渲染 `NModal` 包裹的 `SettingsView`。`SettingsView` 通过 `emit('close')` 通知父组件关闭弹窗。

替代方案是继续调用 `win.openSettings()`，再把设置窗口内部做成模态框外观。这会保留“离开当前窗口上下文”的问题，也不符合用户指定的点击设置按钮弹出模态框要求，因此不采用。

### Decision 2: `SettingsView` 只负责面板内容，不直接负责遮罩

`NModal` 由 `MainView` 负责，`SettingsView` 负责设置面板的头部、标签、内容区和底部栏。这样组件可以同时服务主窗口模态框和独立设置窗口兜底模式；独立模式外层只需要一个全屏容器承载同一面板。

替代方案是在 `SettingsView` 内部直接渲染 `NModal`，但这会让独立路由和主窗口路由都需要额外处理默认显示状态，测试也更难聚焦面板内容，因此不采用。

### Decision 3: 分类标签使用本地 `activeSection`，不是新增路由

设置分类仅属于模态框内部状态，使用 `activeSection` 控制常规、外观、快捷键、隐私安全、存储、关于的展示。切换分类不会写 URL、不会进入 `ui` store，也不会影响主窗口返回路径。

替代方案是为每个设置分类建立主窗口子路由，但这会把短生命周期弹窗状态提升成应用路由状态，收益很低，因此不采用。

### Decision 4: 只读/规划中功能必须可见但不可误操作

隐私安全中的数据库加密、敏感过滤、应用排除名单，以及外观中的强调色、便签纸等原型内容保留为界面项，但使用禁用控件、辅助文案或“规划中”标记表达当前不持久化。用户可以理解未来方向，同时不会以为点击后已经保存。

替代方案是删除所有未落地设置项，只保留现有字段。这样虽然实现最小，但会偏离原型的信息架构，也无法体现设置面板下一阶段的完整组织方式，因此不采用。

### Decision 5: 操作栏表达自动保存，确认按钮只关闭弹窗

现有设置项是即时保存，本次继续保留自动保存。底部“确认”按钮关闭弹窗，“取消”也关闭弹窗但不回滚已保存项，“重置”第一版显示未实现提示，不执行批量重置。

替代方案是引入暂存表单和统一提交，但这会要求对所有设置项增加回滚逻辑，并改变当前 store 的乐观写模型，超出本次范围。

## Risks / Trade-offs

- 模态框内内容较多，低高度窗口可能出现双重滚动 → 面板固定为最大高度，只有内容区滚动，头部和底部保持稳定。
- Naive UI 表单组件默认尺寸偏松散，可能削弱原型的桌面密度 → 使用自定义 CSS 控制面板宽度、行高、分组边界和滚动区域。
- 只读/规划中项可能被用户误解 → 禁用控件旁显示“规划中”或“当前版本只读”，且不绑定保存事件。
- 主窗口测试原本 mock `openSettings` → 改为断言模态框内容出现，并确保不再要求 Tauri 设置窗口。
- 独立设置窗口模式与主窗口模态框共用组件 → `SettingsView` 的关闭按钮只发事件；独立模式由组件提供兜底返回主界面行为或父级忽略关闭事件。

## Migration Plan

1. 先重构 `SettingsView` 为可嵌入设置面板，保留现有状态读取、自动保存和路径复制逻辑。
2. 在 `MainView` 中引入 `NModal` 和本地可见性状态，把设置快捷入口切到模态框。
3. 更新测试，覆盖弹窗打开、分类切换、路径文本渲染和 highlight.js 警告回归。
4. 保留 `App.vue` 对 `ui.mode === 'settings'` 的渲染路径，作为直接打开设置窗口时的兜底。
5. 如出现模态框阻断主窗口操作问题，可临时回退 `MainView` 设置入口为 `win.openSettings()`，不影响 `SettingsView` 面板本身。

## Open Questions

- 后续是否需要把“确认/取消”改成真正的暂存提交模型；当前按既有自动保存策略处理。
- 隐私安全和应用排除名单何时接入后端 schema；当前不在本次变更中承诺。
- 主窗口整体改版完成后，设置入口是否仍放在快捷卡片中，还是迁移到统一标题栏或侧边栏；当前只改现有入口行为。
