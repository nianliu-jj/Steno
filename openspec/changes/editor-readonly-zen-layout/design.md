## Context

当前主窗口编辑页由 `src/views/NoteEditorView.vue` 驱动，核心正文区域始终渲染 `MarkdownEditor`，因此用户看到的是原始 Markdown 输入界面，而不是可切换的阅读态。页脚目前只展示标签和保存信息，没有“只读/编辑”切换，也没有 Zen 入口。与此同时，`src/views/ZenMode.vue` 仍是单栏写作页，没有文档大纲侧栏；`src/components/MainWorkbenchShell.vue` 只支持展开/折叠，不支持拖拽调宽和宽度记忆。

现有代码里已经有两块可复用基础：一是 `useMarkdown()` 已经提供 HTML 渲染能力，可作为阅读态和大纲提取的底层来源；二是 `settings` store + SQLite settings 表已经支持跨启动持久化配置，因此这次的宽度和折叠状态不需要引入新的本地存储机制。需求图中最明确的变化是：编辑页要同时承担“阅读整理”和“继续编辑”两类任务，Zen 要变成同一文档的沉浸写作页面，而不是孤立的极简视图。

## Goals / Non-Goals

**Goals:**

- 让主窗口编辑页同时支持阅读态和编辑态，并保证阅读态显示渲染后的 Markdown 结构。
- 在主窗口编辑页提供固定的模式切换和 Zen 入口，不再依赖外部菜单或其他页面跳转完成写作模式切换。
- 让主窗口编辑页和 Zen 页面都具备基于 Markdown 标题的大纲导航能力，并保持跳转可用。
- 为主工作台侧边栏和 Zen 大纲侧栏补上可拖拽调宽、自动折叠和持久化记忆。
- 压缩主窗口编辑页顶部空白，让文档容器的圆角和正文高度更接近目标原型。

**Non-Goals:**

- 不在本次变更中重写笔记数据模型、自动保存策略或 Markdown 存储格式。
- 不在本次变更中实现评论、协作、同步、多标签页或新的文档分享能力。
- 不把 quicknote 浮窗、置顶便签或设置页整体迁移到新的阅读/大纲布局。
- 不在本次变更中重新设计 Markdown 工具栏语法集合，只调整其在不同模式下的可见性和容器样式。

## Decisions

### Decision 1: 主窗口编辑页采用“单一 Markdown 源 + 本地视图模式”方案

主窗口编辑页继续以 `title/content/tags` 作为单一数据源，但页面内部新增局部的阅读态/编辑态状态。编辑态继续使用 `MarkdownEditor`，阅读态则使用渲染后的 Markdown 文档视图；两者共用同一份未保存内容，不为阅读态创建额外副本。这样可以满足“转译显示”的需求，同时避免在切换模式时引入第二套保存链路。

替代方案是复用 `settings.state.editorMode` 作为全局模式来源，但该字段当前语义是通用编辑器偏好，包含 `split/edit/preview` 三种值，而此次需求明确是编辑页底部即时切换的局部交互。如果直接复用，会把单页操作放大成全局副作用，因此不采用。

### Decision 2: 大纲能力以“解析 Markdown 标题树”为统一来源

主窗口悬浮大纲和 Zen 右侧大纲都从当前 Markdown 正文实时解析标题树，按标题层级生成树状节点。这样主窗口阅读态、编辑态和 Zen 模式都能共享同一份结构定义，也便于测试“点击标题跳转”这类行为。

替代方案是只在渲染后的 DOM 上反向扫描标题节点，但这会让编辑态无法稳定执行跳转，因为 textarea 本身没有语义化标题节点；同时 Zen 与主窗口会出现两套大纲来源，因此不采用。

### Decision 3: 从主窗口编辑页进入 Zen 时使用同窗口路由回退，而不是新开独立窗口

对主窗口编辑页底部的 Zen 按钮，优先使用 `ui` store 记录来源页面并切换到 `mode === 'zen'` 的页面视图；退出 Zen 时按照来源路由回退到进入 Zen 前的页面。这样最符合“关闭 Zen 是退出到进入 Zen 模式的页面”这一要求，也更接近参考图中的单窗口沉浸页面。

替代方案是继续调用 `useWindow().openZen()` 打开独立 Zen 窗口。这个方案会让返回来源页、共享未保存状态和窗口内布局参考图都变得更复杂，因此不作为主窗口编辑页的默认进入方式。

### Decision 4: 顶部圆角、标题留白和悬浮大纲入口一起收敛到新的编辑页容器层

编辑页会新增一个明确的文档表面容器，由它统一控制顶部圆角、正文高度、标题区与正文区之间的间距，以及悬浮大纲按钮的锚点位置。这样能一次性解决工具栏遮挡圆角、标题区空白过大和浮层定位分散的问题。

替代方案是分别在 `MarkdownEditor` 和 `NoteEditorView` 的现有块级样式上打补丁。这会把圆角、顶边背景和浮层定位拆到多个组件里，后续很难维护，因此不采用。

### Decision 5: 侧栏宽度通过现有 settings 表持久化，并对主侧栏定义“图标阈值自动折叠”

主工作台侧边栏、Zen 大纲侧栏的宽度都通过现有 settings 存储体系持久化。主工作台侧边栏在拖拽到“只剩图标列”阈值时自动折叠，并暂时禁用拖拽，只有用户主动展开后才重新允许调宽。这样既满足图示要求，也避免出现极窄状态下持续误拖的问题。

替代方案是把宽度只放在运行时 store 或 `localStorage`。这会与当前已经存在的 SQLite settings 体系平行重复，也无法保持多窗口逻辑一致，因此不采用。

## Risks / Trade-offs

- 编辑态大纲跳转依赖 textarea 定位而非 DOM 锚点 -> 通过统一的标题偏移计算和组件暴露的滚动/选区接口来降低定位误差，并在测试中覆盖长文场景。
- 阅读态与编辑态切换可能造成未保存内容错觉 -> 切换必须基于当前内存态内容渲染，且不得等待数据库 round-trip 后再显示。
- 主侧栏拖拽过程中频繁写 settings 可能带来抖动 -> 使用拖拽过程中的本地状态，结束拖拽后再持久化，避免每个 pointermove 都写数据库。
- Zen 页面加入大纲后可能削弱“无干扰”感 -> 侧栏只显示标题树，不显示主导航、全局搜索或额外工具，保持功能边界清晰。
- 自动折叠阈值如果过高可能误触发 -> 阈值应直接锚定到“只保留图标 rail 宽度”的实际值，并在拖拽句柄上提供清晰视觉反馈。

## Migration Plan

1. 先扩展设置模型和测试，加入侧栏宽度/折叠状态字段，保证后续布局改造有稳定持久化入口。
2. 为主窗口编辑页补上阅读态、底部动作和大纲能力，并在当前 NoteEditorView 中验证模式切换与跳转。
3. 重构 Zen 页面为“正文区 + 大纲侧栏”布局，并把来源路由回退接入 `ui` store。
4. 在 `MainWorkbenchShell` 中实现主侧栏拖拽调宽、图标阈值折叠与恢复；随后补齐 Zen 侧栏宽度记忆。
5. 统一回归主窗口编辑页、Zen 页面、拖拽持久化和应用重启后的布局恢复行为。

## Open Questions

- 当前需求没有要求把阅读/编辑模式跨会话记忆，因此本次默认将其视为页面内局部状态；如果后续产品希望全局记忆，可在下一轮单独定义。
