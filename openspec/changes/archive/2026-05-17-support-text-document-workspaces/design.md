## Context

当前 Steno 的数据层和主页面都默认把所有内容当作同一种 SQLite 笔记处理：`quicknote`、主窗口新建笔记、置顶便签和 Zen 编辑最终都围绕同一条 `notes` 记录运转。这个模型适合 MVP 阶段，但它无法表达“速记文本”和“磁盘文档”两类对象的不同生命周期，也没有工作区、文件夹、分组这些结构化整理能力。

本次需求同时引入了两条不同的内容轴：

- 磁盘轴：工作区 → 文件夹 → 文档
- 全局轴：分组 → 文本

更复杂的是，主页面最终不是单纯切换两套列表，而是按产品要求把“当前工作区/文件夹里的文件夹与文档”和“全局分组与文本”混合显示在同一个中心列表中；右侧栏则只展示当前工作区的文件夹/文档树。这意味着我们不仅要扩展存储模型，还要在浏览上下文、落盘策略、卡片动作、筛选和统计上建立一套一致的合同。

## Goals / Non-Goals

**Goals:**

- 把当前单一笔记模型升级为统一条目索引模型，明确 `workspace`、`folder`、`group`、`document`、`text` 五类对象。
- 让 `text` 和 `document` 使用不同的正文存储后端：文本正文在数据库，文档正文在磁盘文件。
- 为 `text` 建立 `10KB` 的正文 Markdown 字节限制，并在超限时返回精确提示。
- 让主页面支持“当前工作区内容 + 全局分组/文本”的混合显示，同时让右侧栏只展示当前工作区树。
- 明确 `text -> document` 的原地转换流程、文档落点规则、默认分组规则和无工作区时的提示行为。
- 为卡片类型标识、右键动作、底部当前工作区信息、当前页统计和持久化类型筛选建立一致的前端合同。

**Non-Goals:**

- 不在本次变更中重新设计主编辑页的三态写作内核；上一轮“编辑页与 Zen 写作模式”变更仍然独立存在。
- 不在本次变更中实现跨工作区全文文件监控、外部编辑器双向同步或复杂冲突合并。
- 不支持文本同时属于多个分组，也不支持分组属于工作区。
- 不改变现有 Markdown 导出格式，只调整不同内容类型的底层存储与入口行为。

## Decisions

### Decision 1: 采用“统一条目索引 + 正文分后端存储”的模型

数据库层不再只保存 `notes`，而是引入统一条目索引模型，显式区分：

- `workspace`
- `folder`
- `group`
- `document`
- `text`

其中，数据库负责保存所有条目的索引和元信息；正文内容根据类型分流：

- `text.content` 继续存数据库
- `document.content` 只存磁盘文件

数据库中的文档记录仍会保留标题、路径、标签、更新时间、卡片预览和统计需要的缓存字段，用于主列表、搜索、右键菜单和筛选，不要求每次渲染都直接扫描整个工作区文件系统。

替代方案一是“文档完全不进数据库，只靠扫描文件系统拼列表”；这会让混合列表、类型筛选、右键动作和统计都高度依赖即时扫描，复杂度过高。替代方案二是“文档正文双写到磁盘和数据库”；这会引入双份正文一致性问题，违背本次需求对文档磁盘存储的主语义，因此不采用。

### Decision 2: 文本与文档使用同一条目接口，但创建入口和保存规则分流

主窗口“新建笔记”直接创建 `document`，速记浮窗直接创建 `text`。两者对外都表现为可编辑内容条目，但保存规则不同：

- `text` 保存时先计算正文 Markdown 的 UTF-8 字节数；超过 `10KB` 直接返回失败，不写入数据库。
- `document` 保存时直接写入目标文件路径，并同步更新数据库索引缓存。

这样可以最大限度复用现有卡片、列表和编辑入口，同时保持“文本是轻量速记、文档是长期沉淀”的产品边界。

替代方案是保留“新建笔记先创建数据库笔记，再由用户手动导出成文档”。这不能满足“文档由新建笔记直接生成”的新合同，因此不采用。

### Decision 3: 工作区树和全局分组树在结构层分离，在主列表层混合

右侧结构栏只展示当前工作区的文件夹/文档树，用于明确当前磁盘上下文。全局分组树不进入右侧结构栏，而是在中心列表中与当前工作区内容一起显示。

中心列表显示规则固定为：

- 当前工作区/文件夹下的直接文件夹与直接文档
- 全局分组
- 当前全局文本集合（或当前分组上下文下的文本）

这意味着“混合显示”只发生在列表层，不发生在结构归属层。文件夹/文档仍然属于工作区；分组/文本仍然全局独立。这样既满足了你的 `B` 选择，又不会让数据结构本身失真。

替代方案是让右侧栏同时展示工作区树和分组树，或让分组挂到工作区内部。这会直接破坏“分组不受工作区改变而改变”的要求，因此不采用。

### Decision 4: 文档落点由“当前主页面文档上下文”决定，不由分组上下文决定

创建文档和“文本转文档”都使用统一的文档落点解析规则：

- 如果当前主页面处于工作区根目录，落到工作区根目录
- 如果当前主页面已进入某个子文件夹，落到当前子文件夹
- 如果当前没有工作区，先提示选择工作区，选择后再落到工作区根目录
- 如果用户是在某个文件夹卡片上触发右键新建，则优先以该文件夹作为目标目录

分组上下文不会影响文档落点。也就是说，即便用户当前主要在看某个分组里的文本，只要页面没有有效工作区，就不能直接创建文档；必须先补工作区。

替代方案是“每次都弹窗选目标路径”，但会显著增加常用路径的操作成本，因此不采用。

### Decision 5: `text -> document` 采用原地转换，而不是复制转换

“转为文档”会直接把当前 `text` 条目转成 `document` 条目：

- 文本正文写入目标 `.md` 文件
- 条目类型改为 `document`
- 原来的分组归属清空
- 数据库里保留同一条目的新索引形态，而不是额外复制出第二条记录

这样做最符合“文本是临时速记，文档是正式沉淀”的心智模型，也避免用户在列表里看到一条旧文本和一条新文档的重复内容。

替代方案是复制转换，但这会引入内容重复和后续清理问题，因此不采用。

### Decision 6: 默认分组使用系统内置“收件箱”

为了避免速记流程被强制打断，系统会内置一个不可删除的默认分组 `收件箱`。当用户从速记浮窗保存 `text` 且当前没有显式分组上下文时，文本自动进入 `收件箱`。

这条规则和“没有工作区时不能直接创建文档”形成对称：

- `text` 可以没有人工分类，但不能没有分组容器，因此落到 `收件箱`
- `document` 不能没有工作区容器，因此必须先选工作区

替代方案是要求每次速记都先选分组，但这会显著降低速记效率，因此不采用。

### Decision 7: 类型筛选与标签筛选并存，且类型筛选持久化

筛选面板会新增一组独立的类型筛选项：

- 文件夹
- 分组
- 文档
- 文本

这组筛选项和现有内容标签筛选并存，不互相替代。类型筛选状态需要持久化，确保用户下次打开软件时还能恢复上次的主列表显示偏好。

替代方案是把“文档/文本/文件夹/分组”也做成普通标签，但这会把对象类型和内容标签混在一起，既不清晰也不利于统计，因此不采用。

## Risks / Trade-offs

- [统一条目索引会显著重构当前数据库模型] → 通过“保留旧笔记读写适配层、逐步引入新条目接口”来降低迁移风险，并在 OpenSpec 任务中把数据层改造放在第一阶段。
- [文档正文在磁盘、索引在数据库，可能出现不同步] → 把“文件写成功后再更新索引”作为单向提交顺序；失败时保留旧索引并返回明确错误。
- [混合列表可能让用户困惑‘为什么分组/文本不跟着工作区变化’] → 用底部当前工作区显示、卡片类型标识和右侧栏只展示工作区树来强化边界提示。
- [原地转换文本为文档会改变既有 id / 分组归属语义] → 通过规范转换流程和明确 UI 提示，让用户理解这是“升级为文档”而不是“复制导出”。
- [类型筛选、标签筛选、工作区上下文叠加后会让列表条件复杂] → 在前端使用显式浏览上下文对象，把“当前位置”“类型筛选”“内容标签筛选”三类条件拆开处理。

## Migration Plan

1. 先引入统一条目索引和工作区/分组最小数据结构，但保留现有主列表和编辑页入口不变。
2. 再把速记保存链路分流到 `text`，同时补默认分组与 `10KB` 限制。
3. 接着把主窗口“新建笔记”和“转为文档”接到文档落盘流程，并引入工作区选择提示。
4. 然后重构主列表为混合内容视图，补卡片类型标识、底部当前工作区、右侧结构栏和类型筛选。
5. 最后补齐迁移与回归验证，确保现有 quicknote、主编辑页和 Zen 路由不因内容类型升级而回归。

## Open Questions

- 无。当前产品规则已经通过用户确认锁定；实现阶段允许根据代码结构调整表名、字段名或组件拆分，但不得改变已确认的文本大小限制、原地转换、混合列表显示和文档落点规则。
