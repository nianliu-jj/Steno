# 速记板主题同步与标题编辑设计

## 背景

当前速记板视图 `src/components/StickyNote.vue` 已具备独立窗口、正文双击编辑、内容自动保存和样式工具栏，但存在四个直接影响可用性的问题：

- 右上角窗口操作图标颜色过浅，在浅色速记板背景上对比不足，难以辨认。
- 标题始终处于可输入状态，容易误改，也不符合“点击后才允许编辑”的要求。
- 速记板底部仍允许为单张便签设置独立颜色和透明度，这与当前项目的主题表达割裂。
- 底部缺少字数、行数、保存状态等即时反馈，编辑时缺乏确认感。

本次设计目标不是重做速记板窗口，也不是扩展一套完整的新主题设置，而是在现有 Vue + Tauri 结构内，把速记板的主题语义、标题交互和底部状态补齐，并与主工作台保持一致。

## 已确认决策

本次设计基于以下已确认结论：

- 采用“共享主题令牌”方案，而不是局部补丁或完整主题系统重做。
- 速记板完全跟随项目主题，不再允许单独调整颜色或亮度。
- 标题进入编辑后，若未点击保存而直接失焦，应取消本次修改并恢复原标题。
- 标题保存后，笔记列表中的对应标题需要同步更新。

## 目标

- 让速记板头部所有图标在浅色和深色主题下都具备明确对比度。
- 标题默认只读，通过显式的“编辑 / 保存”图标按钮切换编辑状态。
- 标题修改仅在用户明确保存时提交；失焦或取消时恢复原标题。
- 速记板背景、文字、边框和强调色与项目当前主题保持一致。
- 移除速记板独立颜色与亮度控制，仅保留不影响主题一致性的局部设置。
- 在速记板底部右侧显示字数、行数、保存状态。
- 保持现有笔记保存路径与列表同步机制，不引入新的数据模型。

## 不在本次范围

- 不新增“主题主色”设置项；项目当前主色仍以既有工作台强调色为准。
- 不重做 `PinnedWindowConfig` 的数据库结构；已存在的 `color`、`opacity` 字段本次不迁移、不删除。
- 不改变正文双击进入编辑的现有交互。
- 不扩展速记板为完整的富状态栏或新窗口壳层。
- 不修改主列表卡片的布局与交互模型。

## 推荐方案

采用“应用级共享主题语义 + 速记板显式标题编辑”方案。

核心思路：

- 将主工作台当前使用的浅色 / 深色语义颜色抽到共享主题来源，由 `App.vue` 在每个窗口实例上统一注入。
- `StickyNote.vue` 不再依赖便签自身保存的背景色和透明度决定视觉，而是直接消费共享主题 token。
- 标题编辑从“始终可输 + 自动保存”改为“默认只读 + 显式进入编辑 + 显式保存提交 + 失焦恢复”。
- 正文自动保存与标题显式保存拆开，避免标题草稿被 debounce 自动保存误提交。
- 底部左侧仅保留字号控制，底部右侧增加字数、行数、保存状态。

这个方案既能满足“与项目主题保持一致”，又不会把本次需求扩展成一次完整设置系统重构。

## 结构设计

### 1. 共享主题语义

当前 `src/components/MainWorkbenchShell.vue` 在组件内部直接声明：

- `--bg`
- `--surface`
- `--surface-2`
- `--fg`
- `--muted`
- `--faint`
- `--border`
- `--accent`
- `--accent-soft`

这些变量已经构成了事实上的工作台主题语义，但它们只存在于主工作台壳组件中，速记板无法直接复用。

本次设计将主题语义提升到应用层：

- 在 `src/theme/index.ts` 中补充“工作台 / 速记板共用”的语义 token 生成逻辑。
- `App.vue` 根据当前主题模式（`light` / `dark` / `system` 的解析结果）把语义变量挂到窗口根节点。
- `MainWorkbenchShell.vue` 改为消费应用级变量，不再在组件内部维护一套孤立值。
- `StickyNote.vue` 直接读取同一套变量，并在此基础上派生速记板专用变量，如 `--sticky-surface`、`--sticky-fg`、`--sticky-border`、`--sticky-accent`。

这样做的重点不是“让速记板看起来像主工作台”，而是让两者实际读取同一个主题来源。

### 2. 跨窗口主题同步

速记板是独立窗口，当前每个窗口只在挂载时调用一次 `settings.load()`。这意味着如果用户在主窗口修改主题模式，已打开的速记板不会自动同步。

为了满足“与项目主题和亮度保持一致”，本次设计补一条轻量跨窗口同步链路：

- 约定新的应用事件，例如 `steno:theme-mode-changed`。
- 当设置面板成功更新 `themeMode` 后，主窗口向所有窗口广播该事件。
- 所有窗口的 `App.vue` 监听该事件，并更新本地主题模式解析结果。
- 速记板因此可在不重开窗口的情况下切换浅色 / 深色外观。

这里同步的是主题模式，而不是每张便签的自定义样式。由于项目当前主色固定，跟随同一套语义 token 即可自然保持颜色一致。

### 3. 速记板头部与标题交互

`StickyNote.vue` 的标题区从“输入框常驻”改为“两态结构”：

- 默认态：显示标题文本和一个“编辑标题”图标按钮。
- 编辑态：显示标题输入框和一个“保存标题”图标按钮。

行为约束：

- 默认只有点击编辑按钮，标题才进入可编辑状态。
- 进入编辑后，标题使用临时草稿值，不直接写回已保存标题。
- 点击保存按钮或按 `Enter` 才提交新标题。
- 按 `Esc`、输入框失焦或点击其他区域时，取消本次修改并恢复原标题。
- 标题保存成功后退出编辑态，并同步更新本地“已保存标题”。
- 标题保存失败时保留编辑态和草稿内容，让用户可继续修改或重试。

右上角窗口操作按钮：

- 继续保留“取消置顶并关闭”和“隐藏便签”两个动作。
- 图标颜色不再写死为浅灰，而是统一使用主题前景 / 次前景语义色。
- hover / active 状态由主题 token 派生，保证浅色和深色下都清楚可见。

本次不采用运行时读取背景亮度再动态计算图标颜色的方案，因为速记板背景已不再独立可配色，直接使用共享主题前景语义更稳定、维护成本更低。

### 4. 速记板正文与底部区域

正文区保留现有模式：

- 阅读模式显示 Markdown 预览。
- 双击正文进入编辑。
- 编辑完成后通过现有“完成”按钮或双击空白退出正文编辑。

底部区域调整为两段结构：

- 左侧：保留字号控制。
- 右侧：新增统计信息区，显示 `字数 / 行数 / 保存状态`。

移除项：

- 颜色选择器。
- 透明度滑块。

行数定义：

- 使用当前正文内容按换行切分统计。
- 空内容按 `1 行` 显示，避免出现 `0 行` 的反直觉结果。

保存状态文案：

- `未修改`
- `编辑中`
- `保存中`
- `已保存`
- `保存失败`

状态区只暴露用户能理解的结果，不展示内部 debounce 或 store 细节。

## 数据流设计

### 1. 本地状态拆分

速记板至少拆分以下状态：

- `savedTitle`：当前已保存标题。
- `titleDraft`：标题编辑中的临时草稿。
- `titleEditing`：是否处于标题编辑态。
- `content`：正文内容。
- `tags`：标签数组。
- `config`：仍沿用现有 `PinnedWindowConfig`，但视觉上只继续使用尺寸和字号相关字段。

初始化时：

- 从 `db.getNote(noteId)` 读取笔记。
- `savedTitle` 与 `titleDraft` 同时由后端标题填充。
- `content`、`tags`、`config` 按现有逻辑 hydrate。

### 2. 正文自动保存

当前实现中，正文 autosave 监听 `title` 和 `content`。本次需改为：

- 正文自动保存只监听 `savedTitle`、`content`、`tags` 与需保留的配置项。
- `titleDraft` 不参与正文自动保存 payload。

这样可以避免用户刚开始编辑标题时，debounce 自动保存把未确认的标题草稿提交到数据库。

### 3. 标题显式保存

点击标题保存按钮或按 `Enter` 时，按以下顺序执行：

1. 先 flush 现有正文 autosave 队列。
2. 使用 `titleDraft + 当前 content + 当前 tags + 当前 config` 发起一次显式 `notes.saveDraft(...)`。
3. 保存成功后：
   - 更新 `savedTitle`。
   - 让 `titleDraft` 与 `savedTitle` 对齐。
   - 退出编辑态。
4. 保存失败后：
   - 保留编辑态。
   - 保留 `titleDraft`。
   - 状态栏显示保存失败。

由于 `notes.saveDraft()` 成功后会调用 `upsertLocal(saved)`，主列表中的标题会自动同步，无需额外补一条“通知主列表刷新标题”的独立通道。

### 4. 标题取消

以下行为均视为取消修改：

- 输入框失焦。
- 点击标题区外部。
- 按 `Esc`。

取消时执行：

- `titleDraft = savedTitle`
- `titleEditing = false`
- 不触发保存

这与用户确认的“失焦恢复原标题”完全一致。

### 5. 主题同步

主题变化链路设计如下：

1. 设置面板更新 `themeMode`。
2. 写库成功后广播 `steno:theme-mode-changed`。
3. 各窗口的 `App.vue` 接收事件并更新本地主题解析。
4. 根节点 CSS 变量随之更新。
5. `MainWorkbenchShell.vue` 与 `StickyNote.vue` 自动继承新的浅色 / 深色变量。

这条链路保证：

- 主窗口与速记板窗口共享同一主题模式。
- 已打开速记板无需关闭重开即可获得新主题。

## 视觉与交互原则

### 对比度

- 图标、标题、统计信息全部基于主题前景色或弱化前景色表达。
- 可交互图标的 hover / pressed 状态必须进一步提升对比，而不是只依赖透明度微调。

### 一致性

- 速记板不再使用与主工作台分离的黄色便签色。
- 浅色和深色都应沿用同一套语义层级：背景、表面、边框、正文、弱文案、强调色。

### 交互边界清晰

- 标题编辑与正文编辑应被用户清楚区分。
- 标题只有“编辑中”和“已保存”两个明确结果，不再存在“看起来改了、其实已被自动保存”的灰区。

### 反馈简洁

- 底部状态栏只给出高信号信息。
- 字数、行数、保存状态固定出现在右下角，不随正文编辑模式跳动位置。

## 实现落点

### `src/theme/index.ts`

- 提供共享主题语义定义与浅色 / 深色映射。
- 统一导出主工作台和速记板需要的语义颜色。

### `src/App.vue`

- 在根节点注入共享主题 CSS 变量。
- 增加对 `steno:theme-mode-changed` 的监听与清理。
- 保持现有 `themeMode` 优先级逻辑不变。

### `src/views/SettingsView.vue` 或设置写入链路

- 在 `themeMode` 写库成功后广播主题模式变更事件。
- 仅在写入成功后广播，避免其他窗口收到错误状态。

### `src/components/MainWorkbenchShell.vue`

- 改为消费应用级主题变量。
- 删除或收缩组件内硬编码的本地主题定义，避免与速记板出现两套近似值。

### `src/components/StickyNote.vue`

- 将标题输入拆为默认显示态与编辑态。
- 引入 `savedTitle` / `titleDraft` / `titleEditing` 三段式状态。
- 从正文 autosave 监听中移除“标题草稿”。
- 新增标题显式保存与取消逻辑。
- 移除颜色选择器与透明度滑块。
- 新增底部右侧统计信息区。
- 主题颜色改为消费共享 CSS 变量，而不是 `config.color` / `config.opacity`。

  - 底部右侧显示字数、行数、保存状态。

## 错误处理与回归关注点

- 标题保存前必须先 flush 正文 autosave，避免旧 payload 回写旧标题。
- 标题保存失败时不能静默退出编辑态，否则用户会误以为已保存成功。
- 移除颜色 / 透明度控件后，要确认旧数据中的 `PinnedWindowConfig.color`、`opacity` 不会继续影响显示。
- 主题模式跨窗口广播后，要确认非主工作台窗口不会因事件监听重复注册而产生内存泄漏。
- 深色模式下需确认正文预览、Markdown 代码块、引用块和底部分隔线仍保持足够对比。

## 验证方式

- 单元测试：
  - `StickyNote` 的标题编辑、取消、保存与底部状态展示。
  - `App` 的主题模式同步。
- 手动验证：
  - 打开速记板后，右上角图标在浅色 / 深色下均清晰可见。
  - 标题默认不可编辑，只能通过图标按钮进入编辑。
  - 输入新标题后点击空白会恢复原标题，不触发保存。
  - 点击保存后，速记板标题与主列表标题同步更新。
  - 底部不再显示颜色与亮度控件，只保留字号与统计信息。
  - 在设置中切换浅色 / 深色时，已打开的速记板可同步变化。

## 自检结果

- 没有占位项、待补字段或互相冲突的要求。
- 设计范围集中在速记板主题同步、标题编辑和底部状态，没有扩散到数据库结构重构。
- 用户确认的三项关键决策（共享主题令牌、完全跟随项目主题、失焦取消标题修改）都已落实到交互与数据流设计。
- 方案包含必要的跨窗口主题同步，否则“与项目主题保持一致”在多窗口场景下无法成立。
