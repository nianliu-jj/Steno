# 速记板主题同步、标题显式编辑与状态栏设计

## 背景

当前 [StickyNote.vue](D:/Steno/src/components/StickyNote.vue) 已具备独立窗口、正文双击编辑、正文自动保存和底部样式工具栏，但仍有四个直接影响可用性的缺口：

- 右上角图标在浅色便签背景上对比度不足，难以辨认。
- 标题始终可编辑，容易误触，也不符合“点击图标后才允许编辑”的预期。
- 速记板仍允许为单张便签设置独立颜色和透明度，这与当前项目的主题表达割裂。
- 底部缺少字数、行数、保存状态等即时反馈，编辑时缺少确认感。

本次设计目标是在现有 Vue + Tauri 架构内补齐速记板的主题一致性、标题交互和底部状态，不扩展为新一轮完整主题系统重构。

## 已确认决策

- 采用局部改造方案，主改 [StickyNote.vue](D:/Steno/src/components/StickyNote.vue)，必要时补轻量共享主题与跨窗口事件。
- 速记板完全跟随项目当前主题，不再保留单张速记板自己的颜色和亮度设置。
- 标题默认只读，点击标题后的编辑图标才进入编辑态。
- 标题编辑后，点击保存图标、按 `Enter` 或输入框失焦都视为“提交保存”。
- 标题保存成功后，笔记列表中的对应标题需要同步更新。
- 底部右侧状态格式为 `字数 · 行数 · 保存状态`，其中字数只统计正文，不含标题。

## 目标

- 让速记板右上角操作图标在浅色和深色主题下都具备明显对比。
- 让标题进入编辑成为显式动作，而不是默认一直可输。
- 让标题保存和正文自动保存职责分离，避免标题草稿被正文 autosave 误提交。
- 让速记板视觉语言与主工作台当前主题保持一致。
- 在底部右侧持续显示正文字数、正文行数和保存状态。
- 在不调整数据库结构的前提下兼容旧便签配置。

## 不在本次范围

- 不新增“全局主题主色”设置项。
- 不重做 `PinnedWindowConfig` 的数据库结构。
- 不改变正文双击进入编辑的现有模式。
- 不重做主列表卡片的布局和交互。
- 不把速记板升级为新的完整窗口壳层。

## 推荐方案

采用“共享主题语义 + 标题显式编辑 + 轻量跨窗口同步”方案。

核心思路：

- 将主工作台当前稳定使用的主题语义提取为共享来源，速记板不再依赖自身保存的 `color` 和 `opacity` 决定视觉。
- 标题从“常驻输入框 + 自动保存”改成“默认展示文本 + 点击图标进入编辑 + 显式提交保存”。
- 正文继续保留现有 autosave，标题提交走独立保存入口。
- 成功保存标题后广播一条轻量笔记更新事件，主窗口收到后立即更新列表缓存。
- 底部左侧只保留不会破坏主题一致性的局部设置，底部右侧增加统计与保存状态。

这个方案能满足当前用户需求，同时把改动范围控制在前端层。

## 结构设计

### 1. 主题来源统一

当前项目里真正稳定使用的主题语义来自 [MainWorkbenchShell.vue](D:/Steno/src/components/MainWorkbenchShell.vue) 内部的工作台变量，例如：

- `--bg`
- `--surface`
- `--surface-2`
- `--fg`
- `--muted`
- `--faint`
- `--border`
- `--accent`
- `--accent-soft`

而 [src/theme/index.ts](D:/Steno/src/theme/index.ts) 里的紫色 token 目前并不是主界面实际使用的那一套。为了保证“与项目主题色一致”，本次应以工作台现有暖色 `--accent` 体系为准，而不是引入第三套紫色速记板配色。

设计落点：

- 将当前工作台使用的浅色 / 深色语义颜色提取为共享主题映射。
- `App.vue` 在每个窗口实例根节点挂载共享 CSS 变量。
- `MainWorkbenchShell.vue` 和 `StickyNote.vue` 共同消费这一套变量。
- `StickyNote.vue` 只在共享变量之上派生少量速记板专用变量，如 `--sticky-surface`、`--sticky-fg`、`--sticky-border`、`--sticky-accent`。

这样速记板将直接与项目当前主题同源，而不是通过猜测背景亮度临时计算图标颜色。

### 2. 跨窗口主题同步

速记板是独立窗口。当前每个窗口只在挂载时调用一次 `settings.load()`，因此如果用户在主窗口修改主题模式，已打开的速记板不会自动更新。

为了让“保持一致”覆盖已打开窗口，本次补一条轻量事件链路：

- 新增应用事件 `steno:theme-mode-changed`。
- 设置面板成功更新 `themeMode` 后广播该事件。
- 各窗口的 `App.vue` 监听该事件并更新本地主题解析结果。
- 速记板因此可以在不关闭重开的情况下同步浅色 / 深色切换。

这里同步的是主题模式，不是每张便签自己的配置项。

### 3. 速记板头部与标题交互

标题区从“输入框常驻”改为“两态结构”：

- 默认态：展示标题文本和一个“编辑标题”图标按钮。
- 编辑态：展示标题输入框和一个“保存标题”图标按钮。

交互规则：

- 只有点击编辑图标才进入标题编辑态。
- 进入编辑态时，把当前已保存标题复制到草稿状态。
- 点击保存图标、按 `Enter`、或输入框失焦时，均触发标题提交。
- 按 `Escape` 时取消本次标题修改，恢复到已保存标题。
- 标题保存成功后退出编辑态，保存图标恢复为编辑图标。
- 标题保存失败时保留编辑态和草稿内容，底部状态栏显示失败信息。

右上角窗口操作按钮继续保留：

- 取消置顶并关闭
- 隐藏便签

但视觉上需要改为使用共享主题前景色和 hover 背景，不再写死浅灰色，确保浅色和深色下都清晰。

### 4. 正文区与底部布局

正文区保持当前模式：

- 阅读模式显示 Markdown 预览。
- 双击正文进入编辑。
- 点击“完成”或双击空白退出正文编辑。

底部布局调整为三段式：

- 左侧：字号控制。
- 中间：正文编辑态下的“完成”按钮。
- 右侧：统计与保存状态。

移除项：

- 颜色选择器。
- 透明度滑块。

底部右侧状态文案统一为高信号格式，例如：

- `12 字 · 3 行 · 保存中…`
- `12 字 · 3 行 · 已保存 10:23:15`
- `12 字 · 3 行 · 保存失败：...`

其中：

- 字数只统计正文。
- 行数只统计正文。
- 空正文显示 `0 行`。

## 数据流设计

### 1. 本地状态拆分

速记板至少拆分为以下状态：

- `title`：当前已提交标题。
- `titleDraft`：标题编辑草稿。
- `isTitleEditing`：是否处于标题编辑态。
- `content`：正文内容。
- `tags`：标签数组。
- `config`：沿用 `PinnedWindowConfig`，但视觉上只继续消费尺寸和字号相关字段。

初始化时：

- 从 `db.getNote(noteId)` 拉取笔记。
- `title` 与 `titleDraft` 同时由后端标题填充。
- `content`、`tags`、`config` 按现有逻辑 hydrate。

### 2. 正文自动保存

正文自动保存继续使用现有 [useAutosave.ts](D:/Steno/src/composables/useAutosave.ts)。

调整点：

- autosave 监听 `title`、`content`、`tags` 和仍保留的配置项。
- `titleDraft` 不参与 autosave payload。

这样可以保证标题进入编辑后，用户尚未确认的草稿不会被正文防抖保存误提交。

### 3. 标题提交保存

标题提交路径统一为一个显式保存入口，例如 `commitTitleEdit()`：

1. 若不在标题编辑态，直接返回。
2. 把 `titleDraft` 规范化为最终标题值。
3. 先 `flushSave()`，确保正文 autosave 队列清空。
4. 使用 `titleDraft + 当前 content + tags + config` 调用一次 `notes.saveDraft(...)`。
5. 保存成功后：
   - 更新 `title`
   - 让 `titleDraft` 与 `title` 对齐
   - 退出标题编辑态
   - 广播笔记更新事件
6. 保存失败后：
   - 保留标题编辑态
   - 保留 `titleDraft`
   - 底部状态栏展示失败信息

标题输入框的失焦提交也走同一条路径，不单独实现第二套逻辑。

### 4. 标题取消

取消仅由显式 `Escape` 触发。

取消时执行：

- `titleDraft = title`
- `isTitleEditing = false`
- 不触发保存

这里有意不把“失焦”定义为取消，因为用户已确认失焦应视为保存提交。

### 5. 主列表同步

仅依赖当前窗口里的 `notes.saveDraft()` 和 `upsertLocal()` 不足以保证主窗口列表立刻同步，因为速记板与主列表通常位于不同窗口实例、不同 Pinia 容器。

因此本次补一条轻量跨窗口笔记同步事件：

- 新增应用事件 `steno:note-saved`。
- 速记板标题保存成功后，把最新 `note` 作为事件 payload 广播。
- 主窗口在已加载笔记列表时监听该事件，并调用 `upsertLocal(note)` 更新缓存。

这样可以在不整页重载的情况下同步主列表标题。

### 6. 旧配置兼容

当前 `PinnedWindowConfig` 包含：

- `width`
- `height`
- `opacity`
- `color`
- `fontSize`

本次兼容策略：

- `width` / `height` / `fontSize` 继续保留并持久化。
- `color` / `opacity` 不删除、不迁移，但不再参与视觉渲染，也不再在 UI 中暴露。

这样可以避免动到 Rust 端模型和数据库结构。

## 视觉与交互原则

### 对比优先

- 图标、标题文本、统计信息全部依赖主题前景色或弱化前景色语义表达。
- hover / active 状态必须进一步增强可见性，而不是继续使用近白浅灰。

### 同源而非近似

- 速记板不再延续独立黄色纸张视觉。
- 主题色统一跟随主工作台当前语义色，尤其是现有暖色 `--accent`。

### 交互边界清晰

- 标题编辑和正文编辑是两套独立模式。
- 标题只有“进入编辑”“提交保存”“取消恢复”三种明确动作，不混入正文 autosave。

### 反馈固定

- 统计与保存状态固定在底部右侧。
- 不因正文是否处于编辑态而跳变位置。

## 实现落点

### `src/theme/index.ts`

- 扩展为当前工作台实际使用的共享主题语义导出。
- 提供浅色 / 深色两套映射，继续沿用现有暖色强调体系。

### `src/App.vue`

- 在根节点挂载共享 CSS 变量。
- 监听并响应 `steno:theme-mode-changed`。
- 维持现有 `themeMode` 与 `useDark()` 的优先级逻辑。

### `src/views/SettingsView.vue`

- 在主题设置写库成功后广播 `steno:theme-mode-changed`。

### `src/components/MainWorkbenchShell.vue`

- 改为消费共享主题变量，收束组件内部硬编码主题定义。

### `src/components/StickyNote.vue`

- 把标题输入拆成默认显示态和编辑态。
- 新增 `titleDraft` 和 `isTitleEditing`。
- 从底部移除颜色选择器和透明度滑块。
- 在底部右侧渲染字数、行数和保存状态。
- 用共享主题变量替换当前依赖 `config.color` 和 `config.opacity` 的视觉方案。
- 在标题保存成功后广播 `steno:note-saved`。

### 主列表接收侧

- 在主窗口的笔记列表加载链路中监听 `steno:note-saved`。
- 收到事件后调用 `upsertLocal(note)`，立即刷新卡片标题。

## 错误处理与回归关注点

- 标题保存前必须先 flush 正文 autosave，避免旧 payload 覆盖新标题。
- 标题失焦提交失败时，不能静默退出编辑态。
- 移除颜色 / 透明度控件后，要确认旧数据中的 `color`、`opacity` 不会再影响速记板显示。
- 跨窗口事件监听需要成对清理，避免重复注册。
- 深色模式下需要确认正文预览、代码块、引用块、分隔线和底部状态的可读性。
